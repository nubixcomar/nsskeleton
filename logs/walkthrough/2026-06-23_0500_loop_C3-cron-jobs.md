# Walkthrough — Fase C3: cron con jobs internos + lock

## Resumen ejecutivo
El cron ahora puede ejecutar **jobs internos** (callables PHP) además de comandos de
shell, y el despachador usa un **lock** que evita el solapamiento de corridas.

## Cambios realizados
- **`config/jobs.php`**: jobs internos de ejemplo (`demo:ping`, `cache:clear`, `backup:run`).
- **`App\Services\Jobs`**: `all/names/has/run` (captura salida + valor de retorno; sin excepción).
- **`CronRunner`**:
  - `runTask`: si el comando es `job:<nombre>` ejecuta el job; si no, shell.
  - `runDue`: toma un `flock` no bloqueante sobre `storage/cache/cron.lock`; si otra
    corrida lo tiene, se omite (anti-solapamiento).
- **Form de cron**: `datalist` con los jobs disponibles + ayuda (`job:<nombre>`).

## Verificación
- `php -l` OK.
- **Suite**: `php tests/run.php` → **105/105 PASS** (+3 unit `Jobs`, +2 feature
  `CronRunner` con job + lock).
- **Smoke (CLI)**: `Jobs::run('demo:ping')` → `ok | pong @ …`; `run.php` despacha sin
  errores.
- **Feature**: `runTask('job:demo:ping')` → status `success`, `last_output` con "pong",
  fila en `cron_runs`; con el lock tomado, `runDue` devuelve `[]` (omitido).

## Decisiones de diseño
- Convención `job:<nombre>` para distinguir jobs de comandos shell.
- Jobs como callables en config (simple, sin tablas extra); pueden envolver servicios
  (ej. `backup:run` usa `Backup`).
- Lock por archivo (`flock`) liberado en `finally`.
